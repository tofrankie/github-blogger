## 背景

`extension/panels/multi-select-input.ts` 同时承担命令入口、多步表单、状态收集、配置写入和仓库初始化，导致命名和职责都不清晰。当前代码已经有稳定行为，所以这次更适合做结构重整和命名修复，而不是改动初始化结果。

## 目标 / 非目标

**目标：**

- 把初始化流程拆成更清晰的模块边界
- 把入口文件改成更贴近职责的名字
- 保持现有配置项、交互顺序和成功/失败结果不变

**非目标：**

- 不更改命令语义
- 不引入新依赖
- 不调整 GitHub 仓库创建规则

## 设计决策

### 1. 将入口文件重命名为 `setup-flow.ts`

当前文件名只描述了一个输入控件形式，但实际逻辑是完整的初始化流程。改成 `setup-flow.ts` 后，名字能直接反映它的职责。

备选方案：

- 保留原名：成本最低，但会继续误导读者
- 改成 `init.ts`：更短，但语义太泛

### 2. 将通用多步输入能力与初始化编排分离

把 `MultiStepInput` 这类通用交互能力抽成独立文件，初始化编排只保留步骤顺序、状态持久化和结果提示。这样后续如果再加别的设置流程，可以复用这层能力。

备选方案：

- 全部留在一个文件里：阅读还是会继续吃力
- 按步骤拆成多个文件：颗粒度过细，当前收益不高

### 3. 用步骤数组驱动流程顺序

初始化流程的各个步骤应以一个顺序数组来声明，执行器只负责按数组依次调用。这样如果以后只想交换两个步骤的位置，只需要调整数组顺序，不需要改执行逻辑。

### 4. 保持持久化与仓库初始化逻辑在同一条编排链路里

配置写入、缓存失效和仓库创建是同一个用户动作的后半段，拆得太散会让错误处理更难追踪。它们应继续由初始化编排统一控制，但内部可以抽出小函数降低重复。

备选方案：

- 把写配置和建仓库拆到不同服务：抽象过重，不符合当前规模

### 5. 维持现有用户可见行为

输入顺序、返回/取消、已有仓库视为成功、其他错误展示失败原因都保持不变。重构只改变内部组织，不改变用户已经习惯的结果。

### 6. 共享常量放在 `shared/constants.ts`

扩展端和客户端共用的运行时常量应放在 `shared/constants.ts`。`shared/types/` 只保留类型定义；像 `SettingKey`、`ColorMode` 这类类型可以从常量派生，但不应在类型目录里定义 `SETTING_KEY`、`COLOR_MODE` 这样的运行时常量。

## 风险 / 权衡

- [重命名连锁修改] → 需要同步更新导入路径和相关引用，但影响范围很小
- [文件边界增加] → 文件更多了，不过职责更单一，后续更容易维护
- [步骤数组被误改] → 通过把流程执行和步骤声明分离，降低调整顺序时误伤逻辑的风险
- [共享边界混乱] → 通过约定 `shared/constants.ts` 放运行时常量、`shared/types/` 放类型，降低扩展端和客户端引用来源不一致的风险
- [行为偏移] → 通过保持现有流程和消息文案来降低回归风险

## 迁移方案

1. 新建 `extension/panels/setup-flow.ts`
2. 抽出通用多步输入帮助器到独立文件
3. 更新 `extension/index.ts` 的导入和调用
4. 保持现有配置键和值不变，验证初始化流程仍然可用
5. 如需回滚，只需恢复旧文件名和旧导入关系
